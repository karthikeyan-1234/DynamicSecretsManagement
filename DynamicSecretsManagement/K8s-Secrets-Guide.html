using Microsoft.AspNetCore.Mvc;
using Microsoft.Extensions.Options;

namespace DynamicSecretsManagement.Controllers;

[Route("api/[controller]")]
[ApiController]
public class PaymentController : ControllerBase
{
    private readonly StripeOptions _options;
    private readonly StripeOptions _snapshot;
    private readonly IOptionsMonitor<StripeOptions> _monitor;

    public PaymentController(
        IOptions<StripeOptions>         options,
        IOptionsSnapshot<StripeOptions>  snapshot,
        IOptionsMonitor<StripeOptions>   monitor)
    {
        _options  = options.Value;
        _snapshot = snapshot.Value;
        _monitor  = monitor;
    }

    [HttpGet]
    public IActionResult Get() => Ok(new
    {
        StartupValue  = _options.MaxRetries,              // frozen at pod start
        RequestValue  = _snapshot.MaxRetries,             // re-reads per request
        RealTimeValue = _monitor.CurrentValue.MaxRetries  // immediate
    });
}

Approach 01

Environment Variables

Simplest approach. K8s Secret values injected as env vars via envFrom. .NET maps __-delimited keys to nested config sections automatically.

ℹ
Best for initial development. Secret rotation requires a pod restart.

K8s Secret

The double-underscore __ is the .NET convention — StripeOptions__SecretKey maps to StripeOptions:SecretKey.

Approach 02

Volume Mount + Polling Provider

K8s Secret mounted as /app/secrets/appsettings.json. A PhysicalFileProvider with polling detects changes and triggers IConfiguration reload — no pod restart required.

⚠
Why polling, not FileSystemWatcher? K8s updates secret volumes by swapping a symlink (..data → new-timestamp-dir). FileSystemWatcher listens for inotify modify events and silently misses symlink swaps. Polling stats the file on a ~4s timer and detects the change correctly.

Program.cs

The only code change needed. Also fully compatible with Vault Agent (Approach 3) — no further changes when migrating.

csharp — Program.cs▶
using DynamicSecretsManagement;
using Microsoft.Extensions.FileProviders;

var builder = WebApplication.CreateBuilder(args);
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen(c =>
    c.SwaggerDoc("v1", new() { Title = "DynamicSecretsManagement API", Version = "v1" }));

// Layer 1 — base appsettings.json baked into the image
builder.Configuration
    .AddJsonFile("appsettings.json", optional: false, reloadOnChange: true);

// Layer 2 — K8s Secret (or Vault Agent) volume mount
//
// WHY polling? K8s swaps a symlink on secret update rather than
// modifying the file in-place. FileSystemWatcher misses symlink
// swaps entirely. PhysicalFileProvider with UsePollingFileWatcher
// stats the file on a ~4s timer and detects it correctly.
//
// VAULT READY: When Vault Agent is added it writes a real file
// to this same path — the polling provider picks that up too.
// No changes to Program.cs are ever needed for Vault migration.
var secretsFileProvider = new PhysicalFileProvider("/app/secrets")
{
    UsePollingFileWatcher = true,
    UseActivePolling      = true
};

builder.Configuration.AddJsonFile(
    provider: secretsFileProvider, path: "appsettings.json",
    optional: true, reloadOnChange: true);

builder.Services.AddOptions<StripeOptions>()
    .Bind(builder.Configuration.GetSection("StripeOptions"))
    .ValidateDataAnnotations()
    .ValidateOnStart();

var app = builder.Build();
if (app.Environment.IsDevelopment()) { app.UseSwagger(); app.UseSwaggerUI(); }
app.UseAuthorization();
app.MapControllers();
app.Run();

ℹ
Avoid kubectl edit secret — the editor shows base64 values which must be manually encoded/decoded. Re-applying the YAML is cleaner and keeps source of truth in version control.

Limitations

⚠
K8s Secrets are base64-encoded at rest only. No audit trail, versioning, or granular access control. For production, use Approach 3.
Approach 03

HashiCorp Vault Agent Sidecar

Production-grade. Vault stores secrets encrypted with AES-256. A Vault Agent sidecar authenticates using the pod's Kubernetes Service Account identity — zero hardcoded tokens — and writes /app/secrets/appsettings.json to a shared volume. The polling provider from Approach 2 handles the rest.

Developer rotates secret │ vault kv put secret/data/stripe-config MaxRetries=99 ▼ HashiCorp Vault ← http://vault.local (via Kong Gateway) │ Vault Agent sidecar polls every 2–10s ▼ /app/secrets/appsettings.json ← emptyDir shared volume │ PhysicalFileProvider polls filesystem every ~4s ▼ IConfiguration reloads → IOptionsSnapshot / IOptionsMonitor updated

File Map — what to apply and when

★
Key design principle: The Vault auth setup (enabling K8s auth, writing policy/role) contains no secret values and is fully automated via a Kubernetes Job. The only manual step is vault kv put — where actual secret values appear. Everything else is kubectl apply.

kubectl apply

manual (once)

kubectl apply

K8s Job — enables K8s auth, configures it, writes policy + role, enables KV v2. No secret values.

vault-infrastructure.yml→ init + unseal→ vault-rbac.yml→ create root-token secret→ vault-init-job.yml→ vault kv put→ delete root-token secret→ deployment-vault-agent.yml

3.2 — Initialize & Unseal Vault

One-time operation. Always manual — Vault intentionally does not allow init/unseal to be automated as a security measure.

✕
Critical: Save all 5 unseal keys and the Root Token immediately — shown only once. In production use a KMS. For dev, a local text file is fine.
⚠
After every pod restart Vault will be sealed again and requires re-unsealing with 3 of 5 keys. This is intentional.
powershell — init + unseal▶
# Initialize — generates unseal keys and root token (SAVE THE OUTPUT)
kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault operator init

# Unseal — run 3 times, paste a different key each time
kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault operator unseal
kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault operator unseal
kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault operator unseal

# Verify — Sealed should be false
kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault status

3.3 — RBAC Setup (vault-rbac.yml)

Creates the ServiceAccount your app pod will use as its Vault identity, and grants Vault's own SA permission to call the Kubernetes TokenReview API. No secret values — safe to commit.

ResourcePurpose
ServiceAccount: dynamic-secrets-saIdentity used by the app pod and Vault Agent to authenticate with Vault
ClusterRoleBinding: vault-tokenreview-bindingLets Vault call the K8s TokenReview API to verify SA tokens from app pods. Uses built-in system:auth-delegator role — nothing more is granted.
yaml — vault-rbac.yml▶
# ServiceAccount for the app pod
# Name must match:
#   - bound_service_account_names in vault-init-job.yml
#   - serviceAccountName in deployment-vault-agent.yml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: dynamic-secrets-sa
  namespace: default
---
# ClusterRoleBinding — grants Vault (vault:default SA) permission
# to call the K8s TokenReview API to verify app pod SA tokens.
# system:auth-delegator is a built-in role — grants only this.
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: vault-tokenreview-binding
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind:      ClusterRole
  name:      system:auth-delegator
subjects:
  - kind:      ServiceAccount
    name:      default
    namespace: vault

3.4 — Root Token Secret (temporary)

The Vault Init Job needs to authenticate with Vault to configure it. Since Kubernetes auth isn't set up yet (that's what the Job is doing), the Job uses the root token. You store it temporarily as a K8s Secret, the Job reads it, then you delete it.

⚠
This K8s Secret is temporary. Delete it immediately after the init Job completes successfully. It should never remain in the cluster long-term.
ℹ
Why not commit this to YAML? The root token is a high-privilege credential. It must be passed as a runtime command only — never stored in source control. This single kubectl create secret command is the only manual step in the entire Vault setup flow.
powershell — store root token (replace hvs.xxxx with your actual token)▶
# Run this AFTER vault operator init completes
# Replace hvs.YOUR_ROOT_TOKEN_HERE with the actual root token from init output

kubectl create secret generic vault-root-token `
    --from-literal=token=hvs.YOUR_ROOT_TOKEN_HERE `
    -n vault

# Verify it exists
kubectl get secret vault-root-token -n vault

3.5 — Vault Init Job (vault-init-job.yml)

A Kubernetes Job that configures all Vault internals. Runs once, completes, and logs verification output. Contains no secret values — safe to commit.

Step inside JobWhat it doesIdempotent?
Enable K8s auth methodvault auth enable kubernetes✓ Yes
Configure K8s authSets CA cert + reviewer token so Vault can verify pod SA tokens✓ Yes
Write policyRead-only access to secret/data/stripe-config only✓ Yes
Write K8s auth roleBinds dynamic-secrets-sa in default namespace to policy✓ Yes
Enable KV v2vault secrets enable -path=secret kv-v2✓ Yes
ℹ
The Job has an init container that waits until Vault is unsealed before running — so there's no race condition if you apply all YAMLs in sequence. The || true pattern on already-enabled paths makes every step idempotent and safe to re-run.
yaml — vault-init-job.yml▶
# ServiceAccount for the init Job
apiVersion: v1
kind: ServiceAccount
metadata:
  name: vault-init-sa
  namespace: vault
---
apiVersion: batch/v1
kind: Job
metadata:
  name: vault-init
  namespace: vault
spec:
  backoffLimit: 0              # don't retry — fix and re-apply
  ttlSecondsAfterFinished: 600 # clean up pod after 10 minutes
  template:
    spec:
      serviceAccountName: vault-init-sa
      restartPolicy: Never

      # Wait for Vault to be unsealed before configuring it
      initContainers:
        - name: wait-for-vault
          image: hashicorp/vault:1.19.0
          command:
            - /bin/sh
            - -c
            - |
              echo "Waiting for Vault to be unsealed..."
              until vault status 2>&1 | grep -q "Sealed.*false"; do
                echo "  Not ready — retrying in 5s"
                sleep 5
              done
              echo "Vault is unsealed. Proceeding."
          env:
            - { name: VAULT_ADDR, value: "http://vault.vault.svc.cluster.local:8200" }

      containers:
        - name: vault-init
          image: hashicorp/vault:1.19.0
          env:
            - { name: VAULT_ADDR, value: "http://vault.vault.svc.cluster.local:8200" }
            # Root token read from the temporary vault-root-token K8s Secret.
            # You created this manually after unsealing (step 3.4).
            # Delete it after this job completes: kubectl delete secret vault-root-token -n vault
            - name: VAULT_TOKEN
              valueFrom:
                secretKeyRef:
                  name: vault-root-token
                  key: token
          volumeMounts:
            - name: kube-sa
              mountPath: /var/run/secrets/kubernetes.io/serviceaccount
              readOnly: true
          command:
            - /bin/sh
            - -c
            - |
              set -e
              echo "=== Step 1: Enable Kubernetes auth ==="
              vault auth enable kubernetes 2>&1 || true

              echo "=== Step 2: Configure Kubernetes auth ==="
              vault write auth/kubernetes/config \
                kubernetes_host="https://kubernetes.default.svc.cluster.local:443" \
                kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt \
                token_reviewer_jwt=@/var/run/secrets/kubernetes.io/serviceaccount/token

              echo "=== Step 3: Write policy ==="
              vault policy write dynamic-secrets-policy - <<'EOF'
              path "secret/data/stripe-config" {
                capabilities = ["read"]
              }
              EOF

              echo "=== Step 4: Write Kubernetes auth role ==="
              vault write auth/kubernetes/role/dynamic-secrets-role \
                bound_service_account_names=dynamic-secrets-sa \
                bound_service_account_namespaces=default \
                policies=dynamic-secrets-policy \
                ttl=1h

              echo "=== Step 5: Enable KV v2 secrets engine ==="
              vault secrets enable -path=secret kv-v2 2>&1 || true

              echo "=== Verification ==="
              vault auth list
              vault secrets list
              vault policy read dynamic-secrets-policy
              vault read auth/kubernetes/role/dynamic-secrets-role

              echo ""
              echo "=== Init Job complete. ==="
              echo "NEXT: run vault kv put to store your actual secrets (step 3.6)"
              echo "THEN: kubectl delete secret vault-root-token -n vault"

      volumes:
        - name: kube-sa
          projected:
            sources:
              - serviceAccountToken:
                  path: token
                  expirationSeconds: 600
              - configMap:
                  name: kube-root-ca.crt
                  items:
                    - { key: ca.crt, path: ca.crt }

3.6 — Write Actual Secrets (manual — never in YAML)

✕
Never put actual secret values in a YAML file. Even if the YAML is not committed to source control, it can appear in kubectl get job -o yaml, shell history, and CI/CD logs. Always pass secrets interactively or via a secure CI/CD variable injection.
powershell — store secrets in Vault KV▶
# Login with root token first
kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault login

# Write your secrets — this is the ONLY step with real values
kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault kv put secret/stripe-config `
    SecretKey="sk_test_YOUR_REAL_KEY_HERE" `
    PublishableKey="pk_test_YOUR_REAL_KEY_HERE" `
    MaxRetries="25"

# Verify (shows key names and metadata — not the values)
kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault kv get secret/stripe-config

# Delete the temporary root token secret now
kubectl delete secret vault-root-token -n vault
✓
After deleting vault-root-token, no high-privilege credentials remain anywhere in the cluster. Future secret rotation uses the Vault UI at http://vault.local/ui or the vault CLI — never a K8s Secret.

3.8 — Vault Agent Sidecar Deployment

Replaces deployment-volumeMount.yml. The K8s Secret is removed entirely. An init container renders appsettings.json from Vault before the app starts, and the sidecar daemon keeps it updated.

yaml — deployment-vault-agent.yml▶
# ConfigMap — Vault Agent config + appsettings.json template
apiVersion: v1
kind: ConfigMap
metadata:
  name: vault-agent-config
  namespace: default
data:
  agent-config.hcl: |
    auto_auth {
      method "kubernetes" {
        mount_path = "auth/kubernetes"
        config = {
          role = "dynamic-secrets-role"
        }
      }
      sink "file" {
        config = { path = "/app/secrets/.vault-token" }
      }
    }
    vault {
      address = "http://vault.vault.svc.cluster.local:8200"
    }
    template {
      source      = "/vault/templates/appsettings.tmpl"
      destination = "/app/secrets/appsettings.json"
      wait { min = "2s"; max = "10s" }
    }

  appsettings.tmpl: |
    {{- with secret "secret/data/stripe-config" -}}
    {
      "StripeOptions": {
        "SecretKey":      "{{ .Data.data.SecretKey }}",
        "PublishableKey": "{{ .Data.data.PublishableKey }}",
        "MaxRetries":     {{ .Data.data.MaxRetries }}
      }
    }
    {{- end }}
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: dynamic-secrets-app
  labels:
    app: dynamic-secrets-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: dynamic-secrets-app
  template:
    metadata:
      labels:
        app: dynamic-secrets-app
    spec:
      serviceAccountName: dynamic-secrets-sa

      initContainers:
        # Fetches secrets from Vault and renders appsettings.json
        # BEFORE the app container starts. Ensures ValidateOnStart passes.
        - name: vault-agent-init
          image: hashicorp/vault:1.19.0
          args: [agent, -config=/vault/config/agent-config.hcl, -exit-after-auth=true]
          env:
            - { name: VAULT_LOG_LEVEL, value: "info" }
          volumeMounts:
            - { name: vault-agent-config, mountPath: /vault/config }
            - { name: vault-templates,    mountPath: /vault/templates }
            - { name: shared-secrets,     mountPath: /app/secrets }

      containers:
        - name: app
          image: dynamic-secrets-app:v4
          imagePullPolicy: IfNotPresent
          ports:
            - { containerPort: 80 }
          env:
            - { name: ASPNETCORE_ENVIRONMENT, value: "Development" }
            - { name: ASPNETCORE_URLS,         value: "http://+:80" }
          volumeMounts:
            - { name: shared-secrets, mountPath: /app/secrets, readOnly: true }

        # Vault Agent daemon — re-renders appsettings.json on KV version change.
        # Polling provider in Program.cs detects the new file within ~4s.
        - name: vault-agent
          image: hashicorp/vault:1.19.0
          args: [agent, -config=/vault/config/agent-config.hcl]
          env:
            - { name: VAULT_LOG_LEVEL, value: "info" }
          volumeMounts:
            - { name: vault-agent-config, mountPath: /vault/config }
            - { name: vault-templates,    mountPath: /vault/templates }
            - { name: shared-secrets,     mountPath: /app/secrets }

      volumes:
        - name: vault-agent-config
          configMap:
            name: vault-agent-config
            items:
              - { key: agent-config.hcl, path: agent-config.hcl }
        - name: vault-templates
          configMap:
            name: vault-agent-config
            items:
              - { key: appsettings.tmpl, path: appsettings.tmpl }
        - name: shared-secrets
          emptyDir: {}   # shared between init container, sidecar, and app
---
apiVersion: v1
kind: Service
metadata:
  name: dynamic-secrets-app-service
spec:
  type: ClusterIP
  selector:
    app: dynamic-secrets-app
  ports:
    - { port: 80, targetPort: 80 }
