Kubernetes Secrets Management
for .NET 9 Web APIs

Three progressive approaches — from environment variable injection to zero-downtime real-time secret rotation with HashiCorp Vault Agent on Docker Desktop Kubernetes with Kong Gateway.

Section 01

Overview

This guide covers three progressively more capable approaches for managing secrets in a .NET 9 Web API on Kubernetes. Each approach is self-contained and production context is clearly indicated.

01

Environment Variables

K8s Secret injected as env vars via envFrom. Simple but requires pod restart to pick up changes.

02

Volume Mount + Polling

Secret mounted as appsettings.json. Polling provider enables live IConfiguration reload — no restart.

03

Vault Agent Sidecar

Production-grade. Vault Agent writes secrets to shared volume. Rotate in Vault → app reflects within seconds.

Project Setup

The StripeOptions model, controller, and Dockerfile are shared across all three approaches.

StripeOptions.cs

csharp — StripeOptions.cs ▶ 
using System.ComponentModel.DataAnnotations;

namespace DynamicSecretsManagement;

public class StripeOptions
{
    [Required]
    public string SecretKey { get; set; } = string.Empty;

    [Required]
    public string PublishableKey { get; set; } = string.Empty;

    [Range(1, 100)]
    public int MaxRetries { get; set; }
}

PaymentController.cs

Demonstrates all three IOptions lifetime variants so you can observe their different refresh behaviours.

csharp — PaymentController.cs ▶ 
using Microsoft.AspNetCore.Mvc;
using Microsoft.Extensions.Options;

namespace DynamicSecretsManagement.Controllers;

[Route("api/[controller]")]
[ApiController]
public class PaymentController : ControllerBase
{
    private readonly StripeOptions _options;
    private readonly StripeOptions _snapshot;
    private readonly IOptionsMonitor<StripeOptions> _monitor;

    public PaymentController(
        IOptions<StripeOptions>         options,
        IOptionsSnapshot<StripeOptions>  snapshot,
        IOptionsMonitor<StripeOptions>   monitor)
    {
        _options  = options.Value;
        _snapshot = snapshot.Value;
        _monitor  = monitor;
    }

    [HttpGet]
    public IActionResult Get() => Ok(new
    {
        StartupValue  = _options.MaxRetries,              // frozen at pod start
        RequestValue  = _snapshot.MaxRetries,             // re-reads per request
        RealTimeValue = _monitor.CurrentValue.MaxRetries  // immediate
    });
}
InterfaceRefresh BehaviourDI Lifetime
IOptions<T>Never — frozen at pod startupSingleton
IOptionsSnapshot<T>Per HTTP requestScoped
IOptionsMonitor<T>Immediately on file changeSingleton + change callback

Approach 01

Environment Variables

The simplest approach. Kubernetes Secret values are injected directly as environment variables. .NET maps __-delimited keys to hierarchical config sections automatically.

ℹ
Best for simple setups or initial development. Secret rotation requires a pod restart.

K8s Secret

The double-underscore __ is the .NET convention for nested config keys — StripeOptions__SecretKey maps to StripeOptions:SecretKey.

yaml — deployment-minimal.yaml (Secret) ▶ 
apiVersion: v1
kind: Secret
metadata:
  name: stripe-config
type: Opaque
stringData:
  StripeOptions__SecretKey:      "sk_test_initial_1234567890"
  StripeOptions__PublishableKey: "pk_test_initial_1234567890"
  StripeOptions__MaxRetries:     "5"

Approach 02

Volume Mount + Polling Provider

The K8s Secret is mounted as /app/secrets/appsettings.json. A PhysicalFileProvider configured with polling (not FileSystemWatcher) detects changes and triggers IConfiguration reload — no pod restart required.

⚠
Why polling? Kubernetes updates secret volumes by swapping a symlink (..data → new-timestamp-dir). FileSystemWatcher listens for inotify modify events and silently misses symlink swaps. A polling provider stats the file on a ~4s timer and detects the change correctly.

Program.cs — The Only Code Change

Replace the standard secret path AddJsonFile with a PhysicalFileProvider configured for polling. This code is also fully Vault Agent compatible — no further changes needed when migrating to Approach 3.

csharp — Program.cs ▶ 
using DynamicSecretsManagement;
using Microsoft.Extensions.FileProviders;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen(c =>
    c.SwaggerDoc("v1", new() { Title = "DynamicSecretsManagement API", Version = "v1" }));

// Layer 1 — base appsettings.json baked into the image
builder.Configuration
    .AddJsonFile("appsettings.json", optional: false, reloadOnChange: true);

// Layer 2 — K8s Secret volume mount at /app/secrets/appsettings.json
//
// WHY a polling provider?
//   K8s secret volume updates swap a symlink (...data → new timestamp dir).
//   FileSystemWatcher uses inotify 'modify' events — misses symlink swaps.
//   PhysicalFileProvider with UsePollingFileWatcher stats on a timer (~4s)
//   and correctly detects the change.
//
// FUTURE-PROOF: When Vault Agent is added it writes a real file to this same
//   path — the polling provider picks that up too. No changes needed here.
const string SecretMountPath = "/app/secrets";
const string SecretFileName  = "appsettings.json";

var secretsFileProvider = new PhysicalFileProvider(SecretMountPath)
{
    UsePollingFileWatcher = true,   // poll filesystem, not inotify/FSW
    UseActivePolling      = true    // push change tokens proactively
};

builder.Configuration.AddJsonFile(
    provider:       secretsFileProvider,
    path:           SecretFileName,
    optional:       true,           // app starts even if mount is absent
    reloadOnChange: true);

// Bind & validate eagerly — bad secrets fail fast on pod startup
builder.Services.AddOptions<StripeOptions>()
    .Bind(builder.Configuration.GetSection("StripeOptions"))
    .ValidateDataAnnotations()
    .ValidateOnStart();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseAuthorization();
app.MapControllers();
app.Run();

Updating Secrets (Live Reload)

Edit MaxRetries (or any value) in deployment-volumeMount.yml and re-apply. Kubernetes updates only the Secret — no pod restart occurs. The polling provider picks up the change within ~10–15 seconds.

powershell ▶ 
# Edit the secret value in deployment-volumeMount.yml, then:
kubectl apply -f deployment-volumeMount.yml

# Wait ~10-15 seconds, then verify
Invoke-RestMethod -Uri "http://kong-proxy.local/api/Payment" | ConvertTo-Json
ℹ
Avoid kubectl edit secret — the in-editor values are base64-encoded and must be manually encoded/decoded. Re-applying the YAML is cleaner and keeps your source of truth in version control.

Limitations

⚠
K8s Secrets are only base64-encoded at rest — not encrypted unless etcd encryption is separately configured. No audit trail, versioning, or granular access control. For production, use Approach 3.
Approach 03

HashiCorp Vault Agent Sidecar

Production-grade secret management. Vault stores secrets encrypted with AES-256. A Vault Agent sidecar authenticates with Vault using the pod's Kubernetes Service Account identity (no hardcoded tokens) and writes /app/secrets/appsettings.json to a shared emptyDir volume. The polling provider from Approach 2 handles the rest — no code changes needed.

Developer rotates secret │ vault kv put secret/data/stripe-config MaxRetries=99 │ ▼ HashiCorp Vault ← http://vault.local (via Kong Gateway) │ │ Vault Agent sidecar polls every 2–10s for KV version change ▼ /app/secrets/appsettings.json ← written to emptyDir shared volume │ │ PhysicalFileProvider polls filesystem every ~4s ▼ IConfiguration.Reload() triggered │ ├── IOptionsSnapshot → new value on next HTTP request └── IOptionsMonitor → new value immediately via OnChange

3.1 — Vault Infrastructure

Everything in one YAML file — namespace, persistent volumes, Vault config, service, deployment with init container (fixes Docker Desktop hostPath permissions), and Nginx Ingress rule for vault.local.

ℹ
Why vault.local and not a path prefix? The Vault UI makes internal requests to /ui/, /v1/, /sys/ etc. A prefix like /vault would break all those internal redirects. A dedicated hostname (same pattern as kong-ui.local) is the correct solution.

# 6. Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vault
  namespace: vault
  labels:
    app: vault
spec:
  replicas: 1
  selector:
    matchLabels:
      app: vault
  template:
    metadata:
      labels:
        app: vault

      serviceAccountName: default

            runAsUser: 0

            - { name: vault-logs, mountPath: /vault/logs }

          ports:
            - { containerPort: 8200, name: http }
            - { containerPort: 8201, name: cluster }
          command: ["/bin/sh", "-c"]
          args:
            - |
              cp /vault/config-temp/vault.hcl /vault/config/vault.hcl
              chmod 644 /vault/config/vault.hcl
              exec vault server -config=/vault/config/vault.hcl
          env:
            - { name: VAULT_ADDR,       value: "http://127.0.0.1:8200" }
            - { name: VAULT_API_ADDR,   value: "http://vault.vault.svc.cluster.local:8200" }
            - { name: VAULT_LOG_LEVEL,  value: "info" }
          securityContext:
            capabilities:
              add: [IPC_LOCK]
            runAsUser:  100
            runAsGroup: 1000
          volumeMounts:
            - { name: vault-config-temp, mountPath: /vault/config-temp, readOnly: true }
            - { name: vault-config,      mountPath: /vault/config }
            - { name: vault-data,        mountPath: /vault/data }
            - { name: vault-logs,        mountPath: /vault/logs }
          # uninitcode=204 + sealedcode=204 prevent restart-loop while
          # you perform the one-time init + unseal
          livenessProbe:
            httpGet:
              path: /v1/sys/health?standbyok=true&uninitcode=204&sealedcode=204
              port: 8200
            initialDelaySeconds: 60
            periodSeconds: 15
            failureThreshold: 5
          readinessProbe:
            httpGet:
              path: /v1/sys/health?standbyok=true&uninitcode=204&sealedcode=204
              port: 8200
            initialDelaySeconds: 10
            periodSeconds: 5
      volumes:
        - { name: vault-config-temp, configMap: { name: vault-config } }
        - { name: vault-config,      emptyDir: {} }
        - { name: vault-data,        persistentVolumeClaim: { claimName: vault-data-pvc } }
        - { name: vault-logs,        persistentVolumeClaim: { claimName: vault-logs-pvc } }

3.2 — Initialize & Unseal Vault

One-time operation after first deploy. Vault starts completely locked until initialized and unsealed.

✕
Critical: Save all 5 unseal keys and the Root Token immediately — Vault shows them only once. In production use a key management system. For dev, a local text file suffices.
  1. Initialize — generates unseal keys and root token
    powershell▶ 
    kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault operator init
  2. Unseal — run 3 times with 3 different keys
    powershell▶ 
    # Run once for each of 3 different unseal keys
kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault operator unseal
  3. Verify — confirm Sealed = false
    powershell▶ 
    kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault status
# Expect: Initialized = true, Sealed = false
⚠
After every pod restart Vault will be sealed again and requires re-unsealing with 3 of 5 keys. This is intentional — it protects the in-memory encryption keys.

3.5 — Kubernetes Auth Setup

This wires Kubernetes pod identity (Service Account) to Vault access control (Policy + Role). The Vault Agent sidecar uses this to authenticate without any hardcoded tokens.

  1. Enable Kubernetes auth method
    powershell▶
    kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault auth enable kubernetes
  2. Configure auth with CA cert + reviewer token
    powershell▶
    kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") -- vault write auth/kubernetes/config `
    kubernetes_host="https://kubernetes.default.svc.cluster.local:443" `
    kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt `
    token_reviewer_jwt=@/var/run/secrets/kubernetes.io/serviceaccount/token
  3. Grant Vault's ServiceAccount permission to verify tokens (TokenReview)
    powershell▶
    kubectl create clusterrolebinding vault-tokenreview-binding `
    --clusterrole=system:auth-delegator `
    --serviceaccount=vault:default
  4. Create a Vault policy — read-only access to your secret path only
    powershell▶
    'path "secret/data/stripe-config" { capabilities = ["read"] }' | `
    kubectl exec -n vault -i $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") `
    -- vault policy write dynamic-secrets-policy -
  5. Create Kubernetes auth role — binds ServiceAccount to Policy
    powershell▶
    kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") `
    -- vault write auth/kubernetes/role/dynamic-secrets-role `
    bound_service_account_names=dynamic-secrets-sa `
    bound_service_account_namespaces=default `
    policies=dynamic-secrets-policy `
    ttl=1h
  6. Create the ServiceAccount for your app
    powershell▶
    kubectl create serviceaccount dynamic-secrets-sa -n default
  7. Verify
    powershell▶
    kubectl exec -n vault -it $(kubectl get pod -n vault -l app=vault -o jsonpath="{.items[0].metadata.name}") `
    -- vault read auth/kubernetes/role/dynamic-secrets-role

kubectl get serviceaccount dynamic-secrets-sa -n default

3.6 — Vault Agent Sidecar Deployment

Replaces deployment-volumeMount.yml. The K8s Secret is removed entirely. An init container renders appsettings.json from Vault before the app starts (so startup validation passes), and the sidecar daemon keeps it updated thereafter.

yaml — deployment-vault-agent.yml ▶ 
# ConfigMap — Vault Agent config + appsettings.json template
apiVersion: v1
kind: ConfigMap
metadata:
  name: vault-agent-config
  namespace: default
data:
  agent-config.hcl: |
    auto_auth {
      method "kubernetes" {
        mount_path = "auth/kubernetes"
        config = {
          role = "dynamic-secrets-role"
        }
      }
      sink "file" {
        config = {
          path = "/app/secrets/.vault-token"
        }
      }
    }

    vault {
      address = "http://vault.vault.svc.cluster.local:8200"
    }

    template {
      source      = "/vault/templates/appsettings.tmpl"
      destination = "/app/secrets/appsettings.json"
      wait {
        min = "2s"
        max = "10s"
      }
    }

  appsettings.tmpl: |
    {{- with secret "secret/data/stripe-config" -}}
    {
      "StripeOptions": {
        "SecretKey":      "{{ .Data.data.SecretKey }}",
        "PublishableKey": "{{ .Data.data.PublishableKey }}",

  template:
    metadata:
      labels:
        app: dynamic-secrets-app
    spec:
      serviceAccountName: dynamic-secrets-sa

      initContainers:
        # Runs once — fetches secrets from Vault and renders
        # appsettings.json before the app container starts.
        # Ensures StripeOptions.ValidateOnStart() passes.
        - name: vault-agent-init
          image: hashicorp/vault:1.19.0
          args: [agent, -config=/vault/config/agent-config.hcl, -exit-after-auth=true]
          env:
            - { name: VAULT_LOG_LEVEL, value: "info" }
          volumeMounts:
            - { name: vault-agent-config, mountPath: /vault/config }
            - { name: vault-templates,    mountPath: /vault/templates }
            - { name: shared-secrets,     mountPath: /app/secrets }

      containers:
        # Main app — reads from shared emptyDir (read-only)
        - name: app
          image: dynamic-secrets-app:v4
          imagePullPolicy: IfNotPresent
          ports:
            - { containerPort: 80 }
          env:
            - { name: ASPNETCORE_ENVIRONMENT, value: "Development" }
            - { name: ASPNETCORE_URLS,         value: "http://+:80" }
          volumeMounts:
            - { name: shared-secrets, mountPath: /app/secrets, readOnly: true }

        # Vault Agent daemon — watches for KV version changes,
        # re-renders appsettings.json. Polling provider in Program.cs
        # detects the new file within ~4 seconds.
        - name: vault-agent
          image: hashicorp/vault:1.19.0
          args: [agent, -config=/vault/config/agent-config.hcl]
          env:
            - { name: VAULT_LOG_LEVEL, value: "info" }
          volumeMounts:
            - { name: vault-agent-config, mountPath: /vault/config }
            - { name: vault-templates,    mountPath: /vault/templates }
            - { name: shared-secrets,     mountPath: /app/secrets }

      volumes:
        - name: vault-agent-config
          configMap:
            name: vault-agent-config
            items:
              - { key: agent-config.hcl, path: agent-config.hcl }
        - name: vault-templates
          configMap:
            name: vault-agent-config
            items:
              - { key: appsettings.tmpl, path: appsettings.tmpl }
        # emptyDir shared between init container, sidecar, and app
        - name: shared-secrets
          emptyDir: {}

ErrorCauseFix

PVC not foundPVCs missing or wrong namespaceEnsure PV + PVC are in vault namespace with storageClassName: manual

Vault Agent init container stuck at Init:0/1

powershell — get init container logs ▶

ErrorCauseFix
403 permission denied on /v1/auth/kubernetes/loginVault can't call K8s TokenReview APICreate ClusterRoleBinding vault-tokenreview-binding with system:auth-delegator for vault:default (step 3.5.3)
role not foundRole name mismatch in agent config vs VaultVerify role name matches exactly in agent-config.hcl and vault write auth/kubernetes/role/...
Connection refused to VaultWrong Vault addressMust be http://vault.vault.svc.cluster.local:8200 (cross-namespace DNS)

Secrets not reloading after rotation

SymptomCauseFix
All values frozenPolling provider not configuredEnsure UsePollingFileWatcher=true and UseActivePolling=true in Program.cs
Only startupValue frozenExpected — IOptions<T> is by design frozenUse IOptionsSnapshot or IOptionsMonitor if live values needed everywhere
